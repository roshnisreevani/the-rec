@AGENTS.md

## What this is

The Rec — a social app for casual/rec-league athletes (pickup basketball, an office softball team, a friend's first surf session). Friend-group-first, funny-first, explicitly not a performance tracker (Strava) or a serious athlete network. See README.md for full product framing; treat its "Planned" section as stale — Groups, Feed, and Banter described there are already built.

Five tabs (`app/(tabs)/_layout.tsx`): **Profile**, **Groups**, a center **+** button (opens `/create-post` as a modal, not a real tab screen), **Feed**, **Banter**. `app/(tabs)/index.tsx` and `explore.tsx` are unused leftovers from the default Expo template (hidden via `href: null`) — ignore them, don't extend them.

## Architecture

**Stack:** Expo SDK 54, Expo Router 6 (file-based, typed routes), React Native 0.81, React 19, Supabase (Postgres + Auth + Storage), TypeScript strict mode. Path alias `@/*` maps to the repo root.

**Routing & auth gating** (`app/_layout.tsx`): a single root `<Stack>` with three `<Stack.Protected>` groups gated on auth state from `contexts/auth-context.tsx`: password-recovery mode (only `reset-password` reachable), signed-in (everything else), signed-out (only `auth`). Every real screen must be explicitly listed as a `<Stack.Screen>` here — a screen file existing under `app/` is not enough to make it navigable; a missing entry here silently breaks that route. Check this file first whenever a `router.push()` target seems to do nothing.

**Typed routes caveat:** `.expo/types/router.d.ts` is generated by the Expo dev server/Metro and is gitignored. If it's stale (dev server not running) `tsc` will fail on `router.push('/newroute')` for a genuinely-new route even though the route is correct — this is a codegen staleness issue, not a real bug.

**Auth & password recovery:** `contexts/auth-context.tsx` wraps `supabase.auth`, exposes `session`/`isPasswordRecovery`, and handles the password-reset deep link manually (`supabase.auth.exchangeCodeForSession`, listening via `expo-linking`) since `detectSessionInUrl: false` is required for React Native and Supabase-js won't parse incoming URLs itself. The reset-link redirect URL is built with `Linking.createURL(...)` and must be present in the Supabase project's Auth → URL Configuration → Redirect URLs allowlist, or the email link won't come back into the app.

**Supabase:** project ref `dtrjnvbldzyqjtbuceou`. `supabase/migrations/*.sql` is the source of truth for schema — migrations are applied both as files here and live via the Supabase MCP tools, so check the migrations folder (not just `supabase/schema.sql`) for the current shape of a table before assuming it's stale docs.

**Cross-feature logic lives in Postgres triggers, not app code.** This is the dominant pattern in this codebase: `notifications` rows (reactions, comments, follows), Banter group-conversation membership syncing with `group_members`, and auto-adding a group creator as its owner are all done via `SECURITY DEFINER` trigger functions on insert, not by one feature's screen calling into another's. When wiring a new "X happening should notify/affect Y" behavior, follow this pattern (a migration + trigger) rather than importing across feature boundaries in the app code.

**RLS self-reference pattern:** policies that need "is this user a member/owner of this group" cannot express that as a subquery on the same table inside its own policy (Postgres re-applies the policy recursively → error 42P17). The fix used throughout (see the groups and follows migrations) is a `SECURITY DEFINER` helper function (e.g. `is_group_member`, `is_group_owner`) that reads with RLS bypassed, called from the policy instead of an inline subquery.

**Connections → Follows migration:** the app originally had a mutual request/accept "Connections" system; it's been replaced by a one-way Follows model (`lib/follows.ts`, `follows` table — instant, no approval step). `lib/connections.ts` still exists but now only owns what never depended on the request/accept concept: private per-user notes, the private-profile/allow-followers settings, and person search. The legacy `connections` table is kept read-only for historical data (nothing writes to it except `blockUser`'s cleanup of old rows) — don't resurrect request/accept functions there. The Settings toggle for "allow new followers" is still named `allow_connection_requests` in the DB/code — that's a naming relic, not a second system.

**Feed post lifecycle:** posts age out of Feed (or get manually removed) into a private per-user Archive (`app/archive.tsx`, `lib/archive.ts`, `archived_at` column) rather than being hard-deleted. From Archive a post can be reshared (posts a fresh copy), promoted to a "Featured" section on the owner's Profile (deliberately separate from the manually-curated "Pick Your 3"), or deleted for good. `deletePost()` (hard delete) still exists in `lib/posts.ts` but Feed's own delete action calls the soft-delete `archivePost()`.

**Theming:** `contexts/theme-context.tsx` + `constants/style.ts` (`LIGHT_COLORS`/`DARK_COLORS`). Components read colors via `useThemeColors()`, never a static import, so everything reacts to the dark-mode toggle. `constants/style.ts` also defines `FONTS` (custom Urbanist/Permanent Marker faces loaded in `app/_layout.tsx`) — this is currently only applied in Profile, Settings, Auth, and the comments modal; most other screens still use the system font via plain `fontWeight`. Known, not yet rolled out further.

**Error messages:** `lib/error-message.ts`'s `errorMessage(e, fallback)` exists because Supabase-js result errors are plain objects with a `.message` field, not `Error` instances — the common `e instanceof Error ? e.message : fallback` pattern silently swallows the real Postgres/RLS error text. Newer code uses `errorMessage()`; a lot of older screens still use the `instanceof Error` pattern and should be migrated opportunistically, not in a mass rewrite.
